最近在做一个本地优先的 LLM Wiki 项目。它的目标不是再做一个“上传文件然后聊天”的界面，而是把平时散落在 Notion、本地文档、代码会话和聊天记录中的材料，逐步整理成一套可以阅读、追溯、更新和继续推理的知识库。

这件事最直接的灵感来自 Andrej Karpathy 在 2026 年 4 月发布的 [LLM Wiki idea file](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)。我很认同其中一个判断：许多 RAG 系统在每次提问时，都要从原始文档中重新找片段、重新拼接上下文、重新生成答案。即使问题已经问过很多次，新的理解也往往停留在对话窗口里，没有成为下一次思考的基础。

LLM Wiki 想改变的不是模型能力本身，而是知识工作的状态：**让知识从一次次临时检索，变成持续积累的编译产物。**

## 从“每次重新找”到“持续编译”

传统 RAG 的基本路径很熟悉：

```text
原始文档 → 切块与向量化 → 查询时检索片段 → LLM 生成答案
```

它适合大量异构材料的即时召回，也能较好地处理不断增长的文档集合。但它有一个天然特点：知识主要在问答时临时组合。一个需要跨五篇报告、两段会议记录和一次历史决策才能回答的问题，模型必须每次再找一次证据、再做一次综合。

Karpathy 的原始想法则是：在原始资料和最终问答之间，放入一个由 LLM 持续维护的 Markdown Wiki。新材料进入后，模型不是只把它“索引起来”，而是阅读、抽取、更新已有主题页、补足关联、标记矛盾，并把新的综合写回长期存在的页面。

```text
原始材料 → LLM 编译与维护 → 主题化、互相链接的 Wiki → 查询与继续演化
```

这使 Wiki 不只是展示层，而是一个持续积累的中间知识层。Karpathy 用过一个很贴切的比喻：Obsidian 像 IDE，LLM 像程序员，而 Wiki 则像代码库。人主要负责选择值得进入系统的材料、提出问题、检查结果和调整重点；模型承担摘要、归档、链接、维护一致性这些高频而琐碎的工作。

这里“编译”一词很重要。它意味着 Wiki 并不等于原文的拷贝，也不等于聊天记录的堆积；它是从原材料中提炼出的、可被重新阅读和继续修改的结构化产物。

## Karpathy 的三层结构，以及我想补上的一层

Karpathy 在原文中给出了三个核心层次：

```text
Raw sources  原始材料，保持不可变，是事实来源
Wiki         由 LLM 维护的页面、索引、关联和综合
Schema       指导 Agent 如何采集、更新、查询和维护的工作约定
```

这套结构非常简洁。我现在做的项目基本沿着同一个方向，但在 Raw 与 Wiki 之间增加了 `sources/` 层：

```text
Collector
    ↓
raw/       原始或近原始内容，保留 URL、路径、时间和附件
    ↓
sources/   保守清洗后的 canonical source，带元数据、校验和和回溯关系
    ↓
wiki/      概念、项目、人物、决策、问题等可读页面
```

这不是为了多造一个目录，而是为了在“可综合”和“可核验”之间留出边界。

`raw/` 不追求整洁，它保存原始证据；`sources/` 做清理、分类、去重和最少必要的标准化，但避免擅自下结论；`wiki/` 才允许跨材料综合，形成主题页、关系和暂时性的判断。项目中的 `AGENTS.md` 则承担 Karpathy 所说的 schema 角色，明确 Collector、Source Agent 和 Wiki Agent 各自可以做什么、不应该做什么。

对个人笔记来说，Raw 直接进入 Wiki 也许已经足够。但当材料来自会议、项目文档、代码会话和网页，且未来需要回答“这句话从哪里来”“这是原文还是综合判断”时，中间的 canonical source 层会更稳。它让每一条 Wiki 结论都有机会回到 source 章节，再回到 raw package。

## 这个项目试图解决的，不是“存更多资料”

个人知识管理工具从来不缺。真正难的是，资料越来越多之后，系统能否把它们转变成有用的上下文。

我希望这个 LLM Wiki 至少能解决四件事。

### 1. 把会话从一次性消耗品变成可继承的知识

代码 Agent 的会话里常常包含很有价值的东西：为什么选某种架构、某个故障最后是怎样定位的、一个模型为什么失败、哪些命令和验证真正有效。但这些内容通常留在 JSONL、终端输出或聊天历史里，下一次需要时又得重新翻找。

项目为 Claude Code 和 Codex 会话准备了 collector，将会话压缩、分块，并保留项目、时间和原始会话的关联。后续不应该只得到“这次会话的摘要”，而应把其中的项目背景、决策、待解问题和可复用经验编译进 Wiki。

这也是我认为 LLM Wiki 比“对聊天记录做一次总结”更有价值的地方：总结是结束，Wiki 是下一次工作的起点。

### 2. 让回答有出处，而不是只看起来合理

如果一个系统的知识页都由 LLM 写出，最危险的事情是把流畅表述误当成事实。因此本项目把可追溯性放在前面：raw package 保存来源，canonical source 保存清洗后的正文与元数据，Wiki 页面引用 source，查询时再把引用解析回 source 和 raw。

这套设计不保证模型永远不会错，但它要求错误能够被定位和修正。对于不确定、矛盾或信息不足的内容，不应该悄悄并入“已知事实”，而应进入 `wiki/questions/`，把未知也作为知识系统的一部分。

### 3. 让知识库像代码库一样可维护

Karpathy 特别强调 index、log 和 lint。这个思路和软件工程非常接近：不仅要写功能，还要能发现链接断裂、陈旧结论、孤儿页面、重复主题和缺失的交叉引用。

当前项目已经有 Wiki 校验、review 队列和 link graph 的基础能力。它们的意义不是增加流程感，而是承认 Wiki 会老化。随着材料增加，真正需要维护的不是单个页面，而是页面之间的关系、结论和证据链。

所以，一个成熟的 LLM Wiki 不该把“自动写了很多 Markdown”当作成功；更好的目标是：每一轮采集后，系统都能告诉我们哪些判断需要复核、哪些链接失效、哪些问题还没有足够证据。

### 4. 让好的问题和答案也能继续积累

Karpathy 的另一个关键点是：一次高质量查询的结果不应随聊天结束而消失。一次比较、一份决策分析、一个新发现的联系，本身也可以成为 Wiki 的新页面或对旧页面的补充。

这意味着知识库不是单向的“文件进入、答案出来”，而是一个循环：

```text
收集材料 → 编译 Wiki → 基于 Wiki 提问 → 形成新的分析
    ↑                                      ↓
    └──────────── 经 review 后写回 Wiki ───┘
```

不过，写回不能无条件自动发生。尤其是涉及决策、评价、预测或跨材料推断时，应该保留来源、标明它是综合结论，并给人留下 review 的入口。LLM 很适合维护结构，不应该被默认为事实的最终裁判。

## 这不是“反 RAG”，而是为 RAG 增加时间维度

我不认为 LLM Wiki 会替代所有 RAG。

当文档规模很大、变化极快、需要覆盖长尾材料时，检索仍然不可少；当用户的问题要求原始措辞、精确数据或最新外部信息时，也必须回到 source，甚至回到网页或数据库。Karpathy 自己也把 index-first 的方式放在中等规模 Wiki 的语境里，并建议在需要时再补搜索工具。

更合理的理解是：RAG 擅长“此刻从海量材料里找什么”，LLM Wiki 擅长“过去已经学到了什么、哪些关系已经被整理过”。前者解决召回，后者解决积累。一个实用系统很可能同时需要两者：Wiki 承担高信号的长期综合，检索承担证据回查、增量补充和规模扩展。

## 目前所处的位置：流水线已经有了，知识库还要从真实材料长出来

这个项目现在已经具备 collector、source normalization、wiki compilation、review、校验和带 citation 的查询入口等基础流水线。它能够从多个渠道接入材料，也明确了每一层的职责边界。

但我不想把“目录和 Agent 已经搭好”说成“知识库已经建成”。真正有价值的部分还在后面：选择第一批值得沉淀的材料，完成小批量采集，检查每一次清洗与综合是否可信，修复链接和索引问题，然后用真实问题检验 Wiki 是否真的比翻原始聊天更快、更准、更容易复用。

因此，下一阶段最重要的不是继续增加渠道或堆叠复杂 Agent，而是完成一个小而完整的闭环：

```text
少量真实材料
    → raw/source/wiki
    → 人工检查证据链和页面关联
    → 用真实问题查询
    → 将有价值的分析审阅后写回
```

如果这个闭环成立，知识会随着工作而积累；如果不成立，再多的 collector、向量库和自动化也只是在制造更多待整理的文件。

## 人与 LLM 的分工，仍然是这个理念的核心

我最喜欢 Karpathy 这套理念的原因，是它没有把人从知识工作中移除。它只是重新划分了分工。

人负责：

- 选择什么材料值得进入系统；
- 判断哪些问题重要；
- 审核关键综合与决策；
- 为领域设定结构、边界和质量标准。

LLM 负责：

- 读取、清洗、归档和建立链接；
- 更新索引、维护引用和发现矛盾；
- 在已有知识上生成可继续修改的草稿；
- 把繁琐的维护成本降到足够低。

最理想的结果不是拥有一个“替我思考”的黑箱，而是拥有一个能持续整理工作痕迹、保留证据、提醒未知并帮助下一次思考的外部记忆系统。

这正是我想把这个项目做成 LLM Wiki，而不是又一个文件聊天工具的原因。
