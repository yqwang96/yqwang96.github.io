前一阵面试，面试官问我知不知道 DAPO。我只能很老实地说：不太了解。

当时确实有点尴尬。这种题不是那种能临场糊弄过去的类型——你说一句"DAPO 是对 GRPO 的改进"，对方下一句就会问"具体改了什么、为什么这样改"，然后基本就露馅。回家之后我把论文和一些实现说明翻出来认真看了一遍。还没动手跑代码，所以这篇算阅读笔记，不是实战攻略。我也不打算把 DAPO 包装成什么横空出世的新算法，它给我的感觉更像一本"长 CoT 强化学习排雷手册"。

它的起点还是 GRPO：对同一个问题采样一堆回答，把组内 reward 的相对高低转成 advantage，再用带裁剪的 PPO 风格目标更新策略。放在数学、代码这种能自动判对错的任务上，这个思路很优雅。但一旦 reasoning trace 长到几千、上万 token，很多平时会忽略的小细节会突然决定训练还能不能稳得住。

DAPO 就是来处理这些细节的。

## 1. Clip-Higher：别让一条偶然走对的路，太早撞到天花板

GRPO 里 importance ratio 的常见裁剪区间是 `[0.8, 1.2]`。DAPO 没把两边都放宽，只把上界抬到 `1.28`，变成 `[0.8, 1.28]`。

第一眼看到时我心想：就这？但它针对的场景其实很具体：模型偶尔会探索到一条一开始概率很低、最后却答对了的推理路径。在原来的裁剪下，这条路径的正向更新很快就会撞到 `1.2` 的上限，于是它失去了继续被放大的梯度。把上界稍微抬高，等于给这条稀有路径多一点空间，让它从"偶然出现"慢慢变成"下次还来"。

我当时的直觉反应是：那为什么不把下界也一起放宽？后来想，这大概和长 CoT 的 credit assignment 有关。组级 reward 最后要广播到整条回答的每个 token 上；答案错了，不等于前面的每一步都没价值。如果对负 advantage 惩罚得更激进，很容易把前面其实还有用、只是结尾走偏的探索也一起抹掉。

所以我现在记 Clip-Higher 的方式比较粗暴：**对碰运气碰对的探索多推一把，对差点碰对的探索先别急着赶尽杀绝。**

不过要注意一点：`1.28` 这个上界并不是只偏爱低概率 token，它会放宽所有正 advantage token 的更新。把它理解成一个实验验证过的非对称裁剪更贴切，而不是什么理论上最优的概率感知规则。

## 2. Dynamic Sampling：别把 batch 浪费在没有比较价值的问题上

GRPO 的 advantage 来自同一个问题多条回答之间的差异。如果一个组里全对，或者全错，组内 reward 几乎没有方差，大家的 advantage 都接近 0。推理成本已经花了，但 policy gradient 基本就是噪声。

DAPO 的做法很务实：只保留既有正确答案也有错误答案的组；如果有效组不够，就继续采样，直到凑够目标 batch size。

```text
全对：模型已经会了，这一轮没有相对信号。
全错：模型还不会，这一轮同样没有相对信号。
有对有错：才能看出哪种回答值得强化。
```

这有点像在线地把注意力聚焦在模型当前能力边界附近。它不是凭空制造更多梯度，只是提高了一个 batch 里"真正有用梯度"的比例。

但代价也很真实。被扔掉的组已经生成完了，推理成本收不回来。而且训练分布会悄悄偏向那些"有时会、有时不会"的题目——长期完全做不出来的难题可能练得更少，reward 带噪声的样本反而可能被放大。把它叫做工程版的 rejection sampling，比叫它免费午餐更公道。

## 3. Token-Level Policy Gradient Loss：长回答不该天然被稀释

原始 GRPO 常见的一种做法是，先把每条回答内部的 token loss 平均，再对回答平均。这样一来，一条 100 token 的回答和一条 10,000 token 的回答，对最终 loss 的总权重是一样的。

DAPO 改成直接对 batch 里所有有效 token 求平均。每个 token 权重相同，长回答因为 token 多，自然对更新影响更大。

"Token-Level"这个名字其实有点容易误导。它并没有让模型知道"第 317 个 token 是这次答对的关键"，因为同一条回答里的 token 仍然共享同一个回答级 advantage。它改的是 loss 的归约方式，不是 credit assignment 本身。

不过在长 CoT 训练里，这个改动确实有意义。以前长推理虽然要花更多生成成本，却在 loss 里被压成和短回答一样重；改完之后，长推理里的有效模式、重复内容甚至胡说八道都会更充分地进入更新。这到底是不是好事，很大程度上取决于长回答本身的质量——长而正确的 reasoning 会被多学一点，长而拖沓的坏习惯也会被放大。所以它得和长度控制一起看。

## 4. Overlong Reward Shaping：别等撞到墙上才喊"太长了"

长 CoT 一定会设最大生成长度。麻烦的是，回答被截断时不代表一定错了：它可能在胡乱循环，也可能方向完全正确，只是还没来得及输出最终答案。如果一刀切把所有截断都当失败，就会给后一种情况不公平的负反馈。

DAPO 的做法是留一个缓冲区。论文里的设置是：前 16,384 个 token 不做长度惩罚；在最后 4,096 个 token 里，长度 reward 从 0 平滑降到 -1。最终 reward 是任务奖励加上这个逐渐增强的长度惩罚。

这部分是整篇论文里我最快理解的一块。它本质上就是 reward shaping：不是"超过这条线就判死刑"，而是"推理可以长，但解决问题之后请收尾，别一直往长度上限蹭"。

它也让我重新理解截断这件事。DAPO 还讨论过直接过滤掉截断轨迹不学；最终 recipe 更偏向软惩罚。过滤的思路是"这条样本的 reward 不太可信，先不学"；软惩罚的思路是"这条样本还能学，但要让它学会控制长度"。两者解决的问题相近，给出的信号却不一样。

## 把它们串起来看

看完之后，DAPO 在我脑子里不再是一串需要死记硬背的名词，而更像一份常见故障清单：

- 低概率但正确的路径太早撞到更新上限 → Clip-Higher；
- 大量 rollout 因为组内 unanimous 而没有梯度 → Dynamic Sampling；
- 长回答里的 token 被平均得太轻 → Token-Level Policy Gradient Loss；
- 靠近生成上限时奖励变得别扭 → Overlong Reward Shaping。

DAPO 还有一个容易被讨论的基础设定：去掉 reference KL。在数学推理这类可验证任务上，作者希望策略能明显偏离 base model，去探索新的 reasoning 分布，所以不想让 KL 太早把策略拉回去。但这不代表它可以原封不动搬到通用偏好对齐或安全对齐上——那些场景下的 reward hacking、行为漂移和长度偏好都需要重新评估。

## 我的感受

如果只看公式，DAPO 没有推翻 GRPO 的框架：还是 group rollout、relative advantage、带裁剪的更新。它的贡献更像是把采样、裁剪、loss reduction、reward shaping 这些"看起来像实现细节"的东西，系统地拎到台面上。

这恰恰是我读完最有感触的地方。大模型 RL 的难点常常不是写出一个目标函数，而是：一个 batch 里到底有多少有效学习信号，模型是不是过早坍缩成某一种推理模板，长短回答分别被怎样对待，以及被截断的轨迹到底值不值得相信。

这次还停留在理论层面。下一步如果想真正把它学明白，我觉得还是得把 DAPO 的训练循环和 verl 里的实现对照着看——动态补采样到底发生在哪里、loss 是怎么 reduce 的、长度 reward 又是什么时候加进去的。等看过代码再回头想这四个机制，应该会比现在这种"我好像懂了"更踏实一点。
