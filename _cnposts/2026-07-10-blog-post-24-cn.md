很多“让大模型做运筹优化”的方案，最后都会落到一个看似简单的接口：输入题目，输出一个数。但真正困难的部分往往不在最后那一行数字，而在中间过程：变量有没有定义对、约束有没有漏、代码能不能执行、求解器是否真的找到最优解，以及最终答案是否忠实于求解结果。

OR-Agent-R1 是我对这条链路的一次完整实践：让模型把自然语言运筹问题转换成结构化动作，再生成可执行的 PuLP 代码，由 CBC 求解器执行，环境根据执行结果和目标值匹配情况返回奖励。

## 1. 项目想解决什么问题？

传统的问答式评估只看最终答案是否正确；OR-Agent-R1 把任务改写成一个可交互的 Agent episode：

![OR-Agent-R1 的可验证 Agent 闭环](/images/Blog24Fig1.svg)

模型需要依次完成问题抽取、数学建模、代码生成、求解和必要的修正。动作协议采用 XML 标签，核心动作包括 `<extract>`、`<formulate>`、`<solve>`、`<revise>` 和 `<answer>`。其中 `<solve>` 与 `<revise>` 会触发执行器，环境可以观察代码是否运行、solver status 是否为 optimal，以及 objective 是否匹配标准答案。

这使得训练目标从“生成看起来合理的文本”变成“完成一个可以被工具验证的决策过程”。

## 2. 为什么直接用最终答案奖励不够？

如果只有最终的 0/1 答案奖励，模型在一次 episode 中可能经历很多种失败：XML 不合法、动作顺序错误、生成 placeholder、代码无法执行、变量类型错误、约束建错，或者 solver 成功但目标函数不对。所有失败最后都变成同一个 0，模型很难知道下一次应该改哪里。

因此项目把奖励拆成多个可观察层级：

![从稀疏结果到密集可学习信号](/images/Blog24Fig2.svg)

```text
格式正确
  -> 状态转移正确
  -> 产生非空代码
  -> 代码可执行
  -> solver 返回 optimal
  -> objective 与标准答案匹配
```

训练时采用 `solve_format -> solver -> objective` 的 curriculum。先让模型学会稳定地产生合法的工具动作，再把奖励逐步连接到真实求解器，最后强化数学结果正确性。这个顺序的价值不在于让奖励变复杂，而在于让每一个阶段都能回答一个具体问题：Agent 有没有走到正确的工具入口？工具有没有真正执行？执行结果是否完成了任务？

## 3. 工程上最容易误判的地方

早期实验中出现过几乎全零的结果。后来逐项检查才发现，问题并不全是模型能力：

- executor 只接受一种 PuLP import 写法，无法识别模型常见的 `from pulp import *`；
- 代码中使用 `prob` 或 `problem` 时，执行器只寻找 `model`；
- 运行环境缺少 PuLP，导致所有代码在真正求解前就 runtime error；
- `<solve>...</solve>` 这样的 placeholder 没有被单独惩罚；
- 数值答案存在四舍五入和截断，verifier 的 tolerance 过严会把合理解判成错误。

这些问题说明，在 Agent RL 中，环境、执行器和 verifier 本身就是训练系统的一部分。训练前必须先对 parser、状态机、代码提取、solver、reward 和 trace logging 做 canary 测试，否则“模型学不会”和“环境没让它学”很容易混在一起。

## 4. 实验结果：提升来自哪里？

严格主实验使用固定的 train/validation/held-out 划分：1227 条训练样本、153 条验证样本和 154 条最终测试样本。checkpoint 只用 validation 选择，held-out 集只在最后评估一次。

在 tolerance = `5e-3` 时，Qwen2.5-7B-Instruct base model 的 pass@1 为 **64.29%**，经过 LoRA + GRPO 的 RLVR adapter 后提升到 **74.03%**，绝对提升 **9.74 个百分点**。

![Qwen2.5-7B：严格 held-out 集上的 pass@1](/images/Blog24Fig3.svg)

这个提升不是简单的“训练集 reward 变高”。逐题配对分析显示，LoRA 修复了 34 个 base model 失败的样本，同时让 19 个原本成功的样本回归，净提升 15/154；McNemar exact test 的 `p=0.0275`。这也是为什么我更愿意报告配对结果，而不是只比较两个独立平均分。

## 5. 结果背后的真实变化

在开发期的独立对比中，RL LoRA 的 invalid action rate 从 20.4% 降到 4.7%，平均 agent actions 从 5.49 降到 2.805，solver success rate 从 89.0% 升到 92.5%。这说明收益主要来自三类变化：

1. **动作更稳定**：模型更少输出无法被环境接受的动作。
2. **工具链更短**：成功轨迹通常不需要反复尝试，能更快到达 solver。
3. **部分建模错误得到修复**：solver 能返回 optimal 仍不代表建模正确，objective match 才是最终校验。

需要强调的是，开发期 200 样本对比来自当时的训练池，不应被当作严格泛化证据；最终结论以隔离的 held-out 实验为准。

## 6. 这个项目让我得到的几个判断

### Verifier 决定 RLVR 的上限

如果标准答案的精度、变量类型、solver status 或 final answer extraction 有系统偏差，模型会非常稳定地学习这个偏差。RLVR 并不会自动修复一个错误的 verifier。

### 工具调用正确，不等于任务正确

评估至少要拆成协议正确性、代码可执行性、solver 成功率、数学建模正确性和最终答案一致性。只报告其中一个指标，会掩盖真正的瓶颈。

### 小模型也可以从高质量环境中获益

这个实验使用的是 Qwen2.5-7B，而不是超大模型。结果说明，在可验证任务上，先把执行环境、奖励链路和评估方法做扎实，往往比单纯增加训练步数更重要。

## 7. 下一步

目前仍有三个方向值得继续做：对 `base_only` 和 `both_fail` 样本做系统错因分析；用多个随机种子复现训练收益；进一步比较 full curriculum 与 answer-only reward，确认稀疏最终奖励在更长训练中的稳定性。

项目代码和实验脚本已整理在 [OR-Agent-R1](https://github.com/yqwang96/OR-Agent-R1) 中。对我来说，这个项目最重要的产出不是某个单独的 pass@1 数字，而是一套把“语言模型生成”连接到“可执行优化决策”的实验方法。
