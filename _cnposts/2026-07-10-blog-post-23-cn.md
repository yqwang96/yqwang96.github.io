最近在部署 Qwen3-30B-A3B-Thinking 推理模型，需要在两套服务方案之间做选择：一套是 NVIDIA A100 + CUDA + vLLM，另一套是华为昇腾 + vLLM-Ascend。与其只看硬件参数，我直接用同一份 FULL benchmark 压了两套部署，把结果记录下来。这篇博客就是这次实验的整理。

先说明一句：这是**两套服务方案的对比**，不是干净的硬件裸性能对决。两边的框架版本、量化 kernel、权重格式、调度实现和通信后端都不一样，所以下面的差距反映的是整条链路，而不只是芯片本身。

## 1. 结论摘要

在相同压测口径下，A100 4卡方案在所有并发档位上的吞吐更高、延迟更低：

+ 并发 64 时，A100 达到约 **1943.4 output tok/s**，Ascend 8卡约为 **1022.5 output tok/s**——A100 约为 Ascend 的 **1.90 倍**。
+ 并发 64 下 TTFT P95，A100 为 **0.248 s**，Ascend 为 **0.724 s**——A100 的首 token P95 延迟约为 Ascend 的 **34%**。
+ 两套方案在并发 1 到 64 下成功率均为 **100%**，没有 HTTP 500 或请求失败。差异只体现在吞吐和延迟，稳定性没有差别。

两套方案都完成了服务启动、健康检查、模型列表、chat completions、stream 流式输出和 FULL benchmark。

## 3. 测试口径

两套架构采用同口径 FULL benchmark：

+ 并发档位：1、2、4、8、16、32、64。
+ 每个并发档位请求数：200。
+ 输入长度约 512 token，目标输出长度 256 token。
+ 接口：OpenAI-compatible `/v1/chat/completions`，输出模式 `stream=True`。

本轮 token 吞吐没有开启 stream usage，按目标输出长度近似统计（即未启用 `--include-usage`）。由于两边用同一份客户端脚本驱动，结果适合横向比较当前服务方案；如需精确 token 数，可在两边统一开启 `--include-usage` 后复测。

## 5. 结果解读

**吞吐方面。** A100 在所有并发档位上均高于 Ascend，且并发越高差距越明显。并发 64 时 A100 输出吞吐达到 1943.4 tok/s，约为 Ascend 的 1.90 倍。两条曲线在 64 时仍在上升，说明在测试范围内两侧都还没摸到明显的吞吐平台期。

**首 token 延迟（TTFT）。** A100 的 TTFT P95 在并发 1 到 64 之间保持在 0.024 s 到 0.248 s 的较窄区间；Ascend 侧则从 0.140 s 到 0.736 s。A100 的首 token 延迟明显更低，说明当前配置下 prefill、调度和首轮生成链路更快。另外 Ascend 曲线在并发 16 到 32 之间有一次明显跳升，值得关注。

**生成阶段（TPOT）。** A100 的单 token 时间随并发从 0.0102 s/token 上升到 0.0285 s/token；Ascend 则更平——0.0422 到 0.0483 s/token。A100 在低并发时单 token 生成速度优势更大，高并发下 TPOT 上升幅度更大，但即便在并发 64 时仍明显低于 Ascend。

**稳定性。** 两套架构在并发 1 到 64 下均保持 100% 成功率，没有出现服务不可用、HTTP 500 或请求失败。本轮测试中两套部署都可以视为稳定，差异主要体现在吞吐和延迟。

## 6. 对比口径说明

本对比是**服务方案对比**，不是严格的硬件裸性能对比。A100 侧使用 CUDA vLLM 0.10.0、4卡（TP=4）；Ascend 侧使用 vLLM-Ascend 0.19.1rc1、8卡（TP=8）。两边的权重格式、量化 kernel、调度实现、通信后端和软件栈并不相同。

因此，推荐的表述是：在当前部署方案、当前模型包和相同压测脚本下，A100 CUDA vLLM 方案表现出更高吞吐和更低延迟。不要把结果理解成"一张 A100 是一张 Ascend 910B2 的 1.9 倍"这种通用结论。

## 7. 后续建议

+ **精确 token 统计。** 两边统一开启 `--include-usage`，得到更准确的 `completion_tokens` 和 Tok/s。
+ **超过 max_num_seqs 的排队测试。** 测试并发 96、128，观察吞吐平台期和 TTFT 拐点，看排队什么时候开始主导。
+ **长上下文与长输出。** 分别测试 input ≈ 2048/4096 和 output ≈ 512/1024，更接近真实复杂任务场景。
+ **入口并发限制。** 如果考虑线上使用，可以基于 TTFT P95、TPOT P95 和成功率设置入口并发限制，而不是只看峰值 Tok/s。
