过去一段时间，我陆续做了几类 Agent 项目：有面向工业供热场景的问数、异常诊断和控制 Agent，有面向数据分析、代码执行和报告生成的数据科学 Agent，也有把多用户、多项目、Workspace、数据集、模型和 Artifact 管理整合到一起的 Agent 平台。此外，我还在运筹优化任务中尝试过让模型调用求解器，并通过可执行结果验证答案。

这些项目的业务目标差异很大。工业 Agent 更关心安全、权限和现场闭环；数据科学 Agent 更关心文件探查、代码执行、建模和产物管理；平台型 Agent 还要考虑会话、并发、审批、断线恢复和多用户隔离。但做得越多，我越觉得它们底层面对的是同一组问题：**模型应该看到什么、可以调用什么、如何知道下一步该做什么，以及系统如何证明它真的做完了。**

刚开始做 Agent 时，我更关注 ReAct、Planner、Multi-Agent 这些显眼的架构名词。后来真正花时间最多的，反而是 Tool、Skill、Context、状态持久化和安全边界。现在回头看，我认为一个可用 Agent 的最大公约数可以写成：

```text
Agent = 模型决策 + 动态上下文 + 类型化工具 + 执行循环
        + 持久状态 + 权限策略 + 结果验证
```

其中，模型主要负责理解目标和选择下一步；真正决定系统能不能落地的，是模型之外的那部分工程。

## 1. 基础工具集不是越全越好

我最早设计工具时，容易按照已有软件接口逐个映射：文件读取是一个工具，目录遍历是一个工具，数据库每张表一个工具，业务平台每个接口再做一个工具。这样很快就能堆出几十个工具，但模型不一定因此更强，反而会在相似工具之间犹豫，或者调用一个返回大量无关内容的低层接口。

经过几个项目的实践，我更倾向于把基础工具分成四组。

### Workspace 探查工具

```text
list / read / glob / grep
```

它们用于回答最基本的问题：当前有什么文件、文件内容是什么、哪些路径符合条件、某段内容在哪里。对于数据分析、代码和文档类 Agent，这组工具几乎是通用的。

### Workspace 修改与执行工具

```text
write / edit / shell
```

文件修改和命令执行给了 Agent 真正完成任务的能力，但也带来了风险。因此我现在不会只讨论“有没有 Bash”，而会同时确认 Bash 运行在哪里、能访问哪些目录、是否允许联网、CPU 和内存是否受限，以及命令退出后如何保存结果。

对于数据科学 Agent，代码执行应该发生在项目 Workspace 对应的沙箱中；对于工业 Agent，通用 Shell 更不能直接触达控制系统。

### Agent 元能力工具

```text
ask_user / todo / load_skill
```

这类工具不直接完成业务操作，而是帮助 Agent 管理任务。`ask_user` 用于缺少关键信息时暂停，`todo` 用于记录多步骤任务的当前状态，`load_skill` 用于按需加载专业操作规范。

### 产物与资产工具

```text
artifact / datamanager / modelmanager
```

聊天文本不是数据科学任务的全部结果。清洗后的数据、训练脚本、模型文件、评估图表和分析报告应该成为可以持久化、检索和继续使用的资产。把这些能力做成工具后，Agent 的输出才能从“一次回复”变成项目的一部分。

这几组工具不一定全部同时提供给模型。我的经验是，Runtime 可以拥有完整工具目录，但每轮只根据用户权限、项目类型、当前任务和已选 Skill 暴露必要的工具 Schema。

## 2. 领域能力到底做成几个工具？

这是我在工业 Agent 和业务 Agent 中反复遇到的问题。一个领域可能已经有几十个 API 或一套成熟 CLI，到底应该每个操作做一个 Agent Tool，还是只暴露一个大工具？

我的判断标准是下面四件事：

1. 操作的权限等级是否一致；
2. 是否会改变外部状态；
3. 输入参数是否属于同一种结构；
4. 用户和模型是否会把它们理解为同一类动作。

对于数据集管理，使用一个 `datamanager(action, params)` 通常是合理的，因为 list、search、get、profile、register 都围绕同一种资源，返回结构也比较一致。

但对于工业控制，我更倾向于拆成：

```text
heating.query_status
heating.diagnose_anomaly
heating.control_preview
heating.control_execute
```

查询、诊断、预演和真实下发的风险完全不同。特别是 preview 和 execute，即使底层调用的是同一套程序，也应该成为两个独立工具、两条独立审计记录和两个不同的权限点。这样不会因为模型把一个 `dry_run=false` 参数写错，就跨过最重要的安全边界。

另一个心得是：**领域工具既要有原子能力，也要有高层路径。**

查询天气、读取站点状态、查询异常记录和读取历史控制指令属于原子能力；“诊断某个站点的温度异常”则可以是高层工具或 Skill。常见业务路径由高层能力减少调用次数，特殊问题仍允许 Agent 回到原子工具继续探查。

## 3. CLI 很有价值，但不应该成为模型协议

在供热控制和一些专项模型能力中，我使用过 CLI 作为执行入口。后来我认为，CLI 特别适合做领域系统的稳定边界：它可以脱离 Agent 单独测试，可以给运维人员直接使用，也方便放进容器、定时任务和离线部署环境。

但模型不应该直接拼接这种字符串：

```text
control temperature -s 77 -l 低区 -v 45
```

更合适的结构是：

```json
{
  "station_id": "77",
  "loop_name": "低区",
  "target_temperature": 45
}
```

Agent 调用类型化 Tool，Tool 完成参数校验，再由适配器构造 CLI 参数。这样既保留 CLI 的可测试性，也避免模型承担字符串转义、参数顺序和命令安全责任。

我现在更推荐三层结构：

```text
领域核心逻辑
    ↓
Python Service / Library
    ├── CLI：供人工、运维、脚本和测试使用
    └── Agent Tool Adapter：供模型以结构化参数调用
```

CLI 是领域执行边界，Tool 才是模型接口。

## 4. Skill 是操作手册，不是权限系统

在工业异常诊断、每日巡检、数据清洗、时序分析和建模任务中，我逐渐把一些可复用经验整理成 Skill。相比继续扩充 System Prompt，Skill 更适合承载某一类任务的步骤、检查清单、工具使用建议、典型样例和结果解释方式。

但 Skill 不应该决定：

- 用户有没有权限执行控制；
- SQL 是否只读；
- 文件能否写出 Workspace；
- 温度或频率允许调整多少；
- 某个动作是否必须人工审批。

这些约束必须由 Tool、Policy 和 Runtime 强制执行。Skill 可以告诉 Agent“怎样做得专业”，但不能决定“是否允许做”。

### 三级加载方式

```text
Level 1：名称、描述、when_to_use
Level 2：完整 SKILL.md
Level 3：reference、示例、脚本和模板
```

Level 1 可以作为轻量目录进入 Context；Agent 判断相关后调用 `load_skill` 获取完整说明；只有执行到具体步骤时，才继续读取参考文件。这样既保留可发现性，又不会把所有专业知识一次塞进上下文。

### 用户触发与 Agent 主动发现

我现在把 Skill 触发分成四种：

1. **用户显式触发**：用户选择某个 Skill 或使用 Slash Command，本轮直接加载；
2. **Agent 主动触发**：用户目标与 `when_to_use` 明确匹配，Agent 自行加载；
3. **探查后触发**：Agent 先查看文件、Schema 或异常记录，发现任务类型后再加载；
4. **Runtime 触发**：定时巡检、告警或事件任务由调度器直接指定 Skill 和输入。

主动发现不应该只依赖对用户一句话的猜测。比如用户只说“帮我分析这个文件”，Agent 可以先查看文件类型和字段；发现它是多站点时间序列后，再加载时序分析 Skill。这个过程比一开始把所有 Skill 都交给模型更稳定。

## 5. Context Engineering 不是把信息塞满

几个 Agent 做下来，我越来越认同一个判断：上下文窗口不是内存，而是模型这一轮的注意力预算。

一个 128K 的窗口并不意味着应该放入 128K 内容。工具说明、历史消息、Skill、文件内容、数据库结果和执行日志都可能有用，但同时存在时，模型会越来越难判断什么最重要。

我现在倾向于把 Context 分成六层：

```text
1. 稳定系统规则：身份、边界、真实性和安全原则
2. 能力目录：本轮工具 Schema、Skill 元数据、权限
3. 当前任务：用户目标、附件、Todo、待解决问题
4. 项目资产：相关数据集、模型、Artifact 和 Schema 摘要
5. 工作记忆：最近消息、关键工具结果、当前执行步骤
6. 压缩记忆：历史决定、产物引用、失败尝试和未完成事项
```

其中，大型文件、完整查询结果和长日志应该保存在 Workspace 或对象存储中，模型只接收高信号摘要和可继续读取的引用。工具返回“结果已保存到哪个文件、有哪些关键字段、下一步可以如何过滤”，通常比返回几万行原始 JSON 更有用。

### 状态不能只存在于 Context

Context 是模型当前看到的内容，不是系统的可靠状态。运行中的 Todo、审批、已生成资产、任务进度、用户问题和 Checkpoint 都应该持久化到数据库或事件日志。

这个区别在人工审批中尤其明显。如果审批只等待某个内存 Future，而原来的 SSE 连接已经断开，即使用户后来点击同意，Agent 也无法继续。更可靠的方式是保存审批记录、当前消息、待执行工具、已经完成的工具结果和当前位置，再通过新的请求从 Checkpoint 恢复。

### 压缩也有协议边界

不能在模型已经发出 tool call、但 tool result 尚未完整回填时压缩历史。压缩应该发生在一轮工具调用协议完整结束后，并保留用户目标、关键事实、产物、失败原因、待审批项和未解决问题，而不是只生成一段模糊的“对话摘要”。

## 6. 规则驱动的 System Reminder，是很有价值的中间层

在 Context Engineering 之外，我还越来越关注一种更轻量的机制：**Runtime 根据当前事件和规则，动态注入一段短期提醒，引导 Agent 处理眼前最容易出错的事情。**我把它称为 System Reminder。

它既不是一开始就写死的 System Prompt，也不是一套完整的领域 Skill。System Prompt 描述长期稳定的工作原则，Skill 描述完成一类任务的方法，而 System Reminder 只回答一个问题：**此时此刻，有什么信息必须被重新提到注意力最前面？**

Claude Code 给了我不少启发。它公开的机制中，[`CLAUDE.md` 和 Auto Memory](https://code.claude.com/docs/en/memory) 可以提供持久的项目上下文；[Hooks](https://code.claude.com/docs/en/hooks) 则可以在特定事件发生时返回 `additionalContext` 或 `systemMessage`。具体实现不必照搬，但这种“事件发生—规则匹配—上下文注入”的思路很适合进入通用 Agent Runtime。

我现在会这样区分几种能力：

| 机制 | 解决的问题 | 典型生命周期 |
| --- | --- | --- |
| System Prompt | Agent 是谁、长期遵守什么原则 | 每次模型调用 |
| Skill | 一类任务应该怎样完成 | 任务期间按需加载 |
| System Reminder | 当前状态下最需要注意什么 | 下一次调用或当前 Run |
| Tool Result | 外部世界刚刚返回了什么事实 | 当前工具回合 |
| Policy / Guardrail | 哪些行为无论模型如何判断都不能越过 | Runtime 强制执行 |

### Reminder 应该由事件触发，而不是随意插话

System Reminder 最适合处理那些**规则明确、时机重要、有效期很短**的提示。例如：

```text
skill_selected           用户显式选择了 Skill，提醒先加载说明再执行
capability_changed       工具或权限发生变化，提醒重新确认可用能力
context_compacted        上下文刚压缩，重申当前目标、约束和未完成事项
tool_result_truncated    工具结果被截断，提醒通过引用继续分页或检索
tool_failed_repeatedly   同一工具连续失败，提醒停止机械重试并重新诊断
approval_resumed         审批后恢复，提醒不要重复执行已经完成的工具
workspace_file_uploaded  新资产进入 Workspace，提醒先识别类型和可信度
context_budget_warning   注意力预算不足，提醒保存状态并压缩非关键内容
```

以审批恢复为例，Reminder 可以明确告诉模型：“此前三个工具中前两个已经完成，只继续执行第三个待审批动作，不要重复前序写操作。”这类信息如果只埋在一段很长的历史消息里，很容易被忽略；如果永远写进 System Prompt，又没有任何意义。它天然属于事件触发的临时上下文。

### 用规则生成，而不是继续堆 Prompt

这类 Reminder 最好是声明式规则，由 Runtime 根据可信状态生成，而不是由模型自己临时编造：

```yaml
id: resume-after-approval
trigger:
  event: approval_resumed
condition:
  pending_tool_count: "> 0"
scope: next_model_call
priority: high
dedupe_key: approval_id
content_template: |
  本次 Run 从审批点恢复。已完成的工具结果仍然有效，
  只继续 pending_tool_index 指向的动作，不得重复此前的副作用操作。
```

一条可管理的 Reminder 至少应该有来源、触发事件、条件、优先级、作用域、去重键和失效时间。Runtime 在构建下一轮 Context 时，把匹配到的少量高优先级提醒作为独立区块注入；消费一次或条件消失后立即移除，而不是继续沉积在长期对话里。

这会比“发现一次错误，就往 System Prompt 追加一句话”更容易维护。每条规则都能追踪：由什么事件触发、在哪次 Run 中注入、模型是否遵循、是否真的降低了错误率。它因此也很适合接入 Trace 和 Evaluation。

### System Reminder 不能代替硬约束

Reminder 仍然只是在影响模型注意力，并不能提供确定性保证。只允许 `SELECT` 的数据库工具、工业控制的安全阈值、危险动作审批、文件访问范围和租户隔离，都必须由 Policy Engine、状态机和工具代码强制执行，不能依赖一句“请勿违规”。

同时，Reminder 的内容只能来自可信的 Runtime 规则和已验证状态。网页、用户上传文档、数据库文本或普通 Tool Result 中即使出现“忽略之前指令”，也不能被提升成系统级提醒，否则只是给 Prompt Injection 增加了一条高权限通道。

所以我的理想 Context，并不是一份越来越长的超级 Prompt，而是：

```text
稳定的 System Prompt
  + 按需加载的 Skill
  + 来自工具的真实事实
  + 可恢复的 Run State / Memory
  + 规则驱动、短期有效的 System Reminder
```

System Reminder 的价值不在于增加更多规则，而在于让 Runtime 能在正确时机，把已经存在但此刻最关键的规则和状态重新抬到模型眼前。

## 7. 标准 Agent Runtime 应该负责什么

随着项目逐渐产品化，我认为 Runtime 至少需要承担以下职责：

```text
Model Gateway       模型、Provider、Base URL、Context Window、流式协议
Agent Loop          LLM 调用、工具循环、停止条件、取消和最大迭代
Tool Registry       Schema、发现、执行、错误归一化、结果截断
Toolset Resolver    根据权限、项目和 Skill 选择本轮工具
Context Builder     组装规则、历史、状态、资产和用户输入
Reminder Engine     监听事件、匹配规则、去重并注入短期提醒
Compactor           协议安全的上下文压缩
Session / Run Store 会话、Run、事件、并发锁和幂等
Checkpoint          审批、提问、断线和中断恢复
Policy Engine       权限、副作用、风险、审批和限流
Workspace / Sandbox 文件隔离、代码执行和资源限制
Skill Registry      发现、搜索、按需加载、版本和用户范围
Asset Registry      数据集、模型、报告、图表和 Artifact
Event System        SSE / WebSocket 进度和事件
Observability       Trace、Token、延迟、工具错误和任务结果
Evaluation          工具选择、参数正确率、任务成功率和安全评测
```

Multi-Agent 并不是 Runtime 的必选项。很多任务使用一个主 Agent 加一组设计良好的工具已经足够。只有在子任务可以真正并行、需要隔离大量上下文，或者需要独立权限边界时，才值得增加受限的 Sub-Agent。

## 8. System Prompt 是宪法

我在项目中也经历过 System Prompt 越写越长的过程：模型重复查询，就增加一段禁止重试；工具选错，就增加一张工具选择表；报告格式不稳定，再增加报告模板；画图不好看，又加入绘图代码。

短期看，每一段新增内容都在修一个问题；长期看，它会变成工具文档、业务规则、失败分支、输出模板和视觉规范的混合体。规则之间开始冲突，修改一个地方也很难判断会影响哪些任务。

现在我更倾向于只在 System Prompt 中保留稳定原则：

```text
<identity>角色与目标用户</identity>
<scope>可以做什么，不可以做什么</scope>
<truth_policy>必须依据真实工具结果</truth_policy>
<execution_policy>如何探查、行动、检查与停止</execution_policy>
<tool_policy>跨工具的通用选择原则</tool_policy>
<skill_policy>何时加载 Skill，以及指令优先级</skill_policy>
<workspace_policy>输入、中间结果和输出目录规则</workspace_policy>
<safety_policy>权限、敏感数据和危险操作原则</safety_policy>
<completion_contract>完成、失败和产物如何向用户说明</completion_contract>
```

每个工具的参数放在 Tool Schema，领域流程放在 Skill，严格约束放在代码，项目资产通过动态 Context 注入，长篇参考知识通过检索获取。System Prompt 只负责告诉 Agent 它是谁、怎样工作、哪些边界不可跨越。

对于那些只在特定状态下重要的提醒，则交给 Reminder Engine 动态注入，不再为了一个局部故障场景继续膨胀全局 Prompt。

## 9. 几个让我反复踩坑的反模式

### 用 Prompt 修所有问题

如果问题本质是参数校验、权限、超时、重试或状态恢复，继续加 Prompt 通常只是在推迟修复。

### 把底层 API 原样变成工具

人类开发者容易理解几十个资源接口之间的关系，模型未必可以。Agent Tool 应该面向任务，而不只是面向已有 API。

### 把所有 Skill 全文预加载

这样既浪费 Context，也让多个领域流程互相干扰。元数据预加载、正文按需读取更合理。

### 让工具返回过多原始数据

大结果应保存并返回引用；模型上下文中只保留本轮决策需要的信息。

### 把“代码存在”当成“能力在线”

某个 Tool 类、审批模块或 CLI 已经写好，不代表当前运行入口已经注册，也不代表用户有权限使用。Capability Registry 应该明确区分：已实现、已配置、当前启用、当前用户可用。

### 用 Multi-Agent 掩盖工具和上下文问题

如果单 Agent 连工具边界、状态和完成条件都没有设计清楚，拆成多个 Agent 往往只会增加通信和调试成本。

## 10. 我目前更认可的 Agent 形态

如果现在重新设计一套通用 Agent，我会默认采用：

```text
一个主 Agent
  + 一组少而清晰的基础工具
  + 按项目动态加载的领域工具
  + Level 1/2/3 渐进式 Skill
  + 项目级 Workspace 和资产管理
  + 事件化 Run State 与 Checkpoint
  + 规则驱动的短期 System Reminder
  + Tool/Action 级权限和审批
  + 受控沙箱与真实结果验证
```

高风险、确定性强的流程，例如工业控制，不交给模型自由编排，而是让模型负责理解意图、收集证据和生成建议，真正执行仍进入确定性状态机、安全校验和人工审批。开放式的数据分析、代码执行和研究任务，则可以让 Agent 在沙箱中拥有更多自主性。

做完这些项目后，我对 Agent 的理解已经从“能不能让模型调用工具”，转向了另一个问题：**怎样让模型在有限上下文和明确边界内，持续获得真实反馈，并把一次不确定的生成过程变成可观察、可恢复、可验证的系统执行。**

这可能也是我目前认为 Agent 工程最值得长期积累的部分。
