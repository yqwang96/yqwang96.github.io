Recently I have been deploying the Qwen3-30B-A3B-Thinking reasoning model and needed to choose between two serving stacks: an NVIDIA A100 + CUDA + vLLM setup, and a Huawei Ascend + vLLM-Ascend setup. Rather than relying on datasheet numbers, I ran the same FULL benchmark against both deployments and recorded the results. This post is a tidy summary of that experiment.

One note up front: this is a **comparison of two serving solutions**, not a clean bare-metal hardware shootout. The two sides differ in framework version, quantization kernels, weight format, scheduler and communication backend, so the gap below reflects the whole stack rather than the chips alone.

## 1. TL;DR

Under the same benchmark, the A100 4-card stack delivers higher throughput and lower latency across every concurrency level tested:

+ At concurrency 64, A100 reaches about **1943.4 output tok/s**, versus **1022.5 output tok/s** for Ascend 8-card — roughly **1.90×**.
+ TTFT P95 at concurrency 64 is **0.248 s** on A100 vs **0.724 s** on Ascend — A100's first-token P95 is about **34%** of Ascend's.
+ Both stacks achieved a **100% success rate** from concurrency 1 all the way to 64, with no HTTP 500s or failed requests. The difference is purely throughput and latency, not stability.

## 3. Benchmark setup

Both stacks were measured with the same FULL benchmark:

+ Concurrency levels: 1, 2, 4, 8, 16, 32, 64.
+ 200 requests per concurrency level.
+ Input length ≈ 512 tokens, target output length = 256 tokens.
+ Endpoint: OpenAI-compatible `/v1/chat/completions`, with `stream=True`.

Token throughput in this round was computed from the target output length rather than the streamed `usage` field (i.e. `--include-usage` was not enabled). Since the same client script drives both sides, the numbers are valid for a side-by-side comparison of the two deployments. For exact token counts, both sides should be re-run with `--include-usage`.

## 5. What the numbers say

**Throughput.** A100 leads at every concurrency level, and the gap widens as concurrency grows. At concurrency 64 the A100 stack sustains 1943.4 tok/s, about 1.90× the Ascend stack. Both curves are still rising at 64, so neither side has clearly hit its saturation point within the tested range.

**First-token latency (TTFT).** A100's TTFT P95 stays in a tight band from 0.024 s to 0.248 s across concurrency 1–64, while Ascend moves from 0.140 s to 0.736 s. The much lower first-token latency on A100 suggests a faster prefill, scheduling and first-decode path under the current configuration. The Ascend curve also jumps sharply between 16 and 32 concurrent requests, which is worth watching.

**Generation stage (TPOT).** A100's per-token time rises from 0.0102 s/token to 0.0285 s/token as concurrency increases, whereas Ascend is flatter — 0.0422 to 0.0483 s/token. A100's advantage is largest at low concurrency; at high concurrency its TPOT grows faster, but even at 64 it remains well below Ascend.

**Stability.** Both deployments held a 100% success rate from concurrency 1 to 64, with no service unavailability, HTTP 500s or failed requests. For this round both can be considered stable; the meaningful differences are throughput and latency.

## 6. A note on the comparison

This is a comparison of **serving solutions**, not a controlled bare-metal hardware benchmark. The A100 side runs CUDA vLLM 0.10.0 on 4 cards (TP=4); the Ascend side runs vLLM-Ascend 0.19.1rc1 on 8 cards (TP=8). The weight format, quantization kernels, scheduler implementation, communication backend and software stack all differ.

So the careful statement is: under the current deployment, current model package and identical load-testing script, the A100 CUDA vLLM solution shows higher throughput and lower latency. It should not be read as "an A100 is 1.9× an Ascend 910B2" in general.

## 7. Follow-ups

+ **Exact token accounting.** Re-run both sides with `--include-usage` to get accurate `completion_tokens` and tok/s.
+ **Beyond max_num_seqs.** Test concurrency 96 and 128 to find the throughput plateau and the TTFT knee where queuing starts to dominate.
+ **Longer context and output.** Add runs with input ≈ 2048/4096 and output ≈ 512/1024, which is closer to real complex workloads.
+ **Production entry limits.** If either stack goes online, gate admission on TTFT P95, TPOT P95 and success rate rather than chasing peak tok/s.
