Over the past while I built several kinds of Agent projects: question-answering, anomaly-diagnosis and control Agents for industrial district heating; data-science Agents for analysis, code execution and report generation; and an Agent platform that folds multi-user, multi-project, Workspace, dataset, model and Artifact management into one system. I also tried, in operations-research tasks, letting the model call solvers and verifying answers against executable results.

The business goals of these projects differ wildly. Industrial Agents care about safety, permissions and closing the loop on-site; data-science Agents care about file exploration, code execution, modeling and artifact management; platform Agents additionally worry about sessions, concurrency, approvals, reconnect recovery and multi-user isolation. But the more I built, the more I felt they all face the same underlying set of problems: **what the model should see, what it may call, how it knows what to do next, and how the system proves it actually finished.**

When I first started building Agents, I paid more attention to conspicuous architecture terms like ReAct, Planner and Multi-Agent. What I ended up spending the most time on, though, was Tools, Skills, Context, state persistence and safety boundaries. In hindsight, I think the greatest common denominator of a usable Agent can be written as:

```text
Agent = model decisions + dynamic context + typed tools + execution loop
        + persistent state + permission policy + result verification
```

The model is mainly responsible for understanding the goal and choosing the next step; what actually determines whether the system can land in production is the engineering outside the model.

## 1. A bigger base toolset is not a better one

When I first designed tools, I mapped existing software interfaces one by one: one tool for reading files, one for listing directories, one tool per database table, one more per business-platform API. You can pile up dozens of tools very quickly that way, but the model doesn't necessarily get stronger — it hesitates between similar tools, or calls a low-level interface that returns a flood of irrelevant content.

After a few projects, I now prefer to split base tools into four groups.

### Workspace exploration tools

```text
list / read / glob / grep
```

They answer the most basic questions: what files exist, what's in a file, which paths match a pattern, where a piece of content lives. For data-analysis, coding and document Agents, this group is nearly universal.

### Workspace mutation and execution tools

```text
write / edit / shell
```

File edits and command execution give the Agent the ability to actually finish tasks, but they also bring risk. So I no longer just ask "is there a Bash tool" — I also confirm where Bash runs, which directories it can reach, whether network access is allowed, whether CPU and memory are capped, and how results are persisted after the command exits.

For a data-science Agent, code execution should happen in the sandbox attached to the project Workspace; for an industrial Agent, a general-purpose shell must never reach the control system directly.

### Agent meta-capability tools

```text
ask_user / todo / load_skill
```

These tools don't perform business operations directly; they help the Agent manage the task. `ask_user` pauses when key information is missing, `todo` records the current state of a multi-step task, and `load_skill` loads professional operating procedures on demand.

### Artifact and asset tools

```text
artifact / datamanager / modelmanager
```

Chat text is not the whole result of a data-science task. Cleaned datasets, training scripts, model files, evaluation charts and analysis reports should become assets that can be persisted, retrieved and reused. Once these capabilities exist as tools, the Agent's output can turn from "one reply" into part of the project.

These tool groups don't all have to be exposed to the model at once. My experience: the Runtime can own the full tool catalog, but each turn should expose only the tool schemas necessary for the user's permissions, the project type, the current task and the selected Skills.

## 2. How many tools should a domain capability become?

This is a question I hit repeatedly in industrial and business Agents. A domain may already have dozens of APIs or a mature CLI — should each operation become its own Agent Tool, or should you expose one big tool?

My criteria are these four things:

1. whether the operations share the same permission level;
2. whether they mutate external state;
3. whether the input parameters share one structure;
4. whether users and the model would understand them as the same kind of action.

For dataset management, a single `datamanager(action, params)` is usually reasonable, because list, search, get, profile and register all revolve around the same resource and return fairly consistent structures.

But for industrial control, I prefer to split:

```text
heating.query_status
heating.diagnose_anomaly
heating.control_preview
heating.control_execute
```

Querying, diagnosing, previewing and actually issuing a control command carry completely different risk. Preview and execute in particular — even if they call the same underlying program — should be two independent tools, two independent audit records and two distinct permission points. That way, the most important safety boundary can't be crossed just because the model wrote a `dry_run=false` parameter wrong.

Another lesson: **domain tools need both atomic capabilities and high-level paths.**

Querying weather, reading a station's status, querying anomaly records and reading historical control commands are atomic capabilities; "diagnose the temperature anomaly at a given station" can be a high-level tool or a Skill. High-level capabilities reduce the number of calls on common business paths, while unusual problems still let the Agent drop back to atomic tools and keep exploring.

## 3. CLIs are valuable, but they shouldn't be the model's protocol

In heating control and some specialized model capabilities, I used CLIs as execution entry points. I later came to believe that a CLI is especially well suited as the stable boundary of a domain system: it can be tested without the Agent, used directly by operators, and easily dropped into containers, cron jobs and offline deployments.

But the model should not be directly concatenating strings like:

```text
control temperature -s 77 -l low-zone -v 45
```

A more appropriate structure is:

```json
{
  "station_id": "77",
  "loop_name": "low zone",
  "target_temperature": 45
}
```

The Agent calls a typed Tool, the Tool validates the parameters, and an adapter constructs the CLI arguments. This keeps the CLI's testability while relieving the model of responsibility for string escaping, argument order and command safety.

I now recommend a three-layer structure:

```text
Domain core logic
    ↓
Python Service / Library
    ├── CLI: for humans, operators, scripts and tests
    └── Agent Tool Adapter: for the model to call with structured parameters
```

The CLI is the domain's execution boundary; the Tool is the model's interface.

## 4. A Skill is an operating manual, not a permission system

Across industrial anomaly diagnosis, daily inspection routines, data cleaning, time-series analysis and modeling tasks, I gradually organized reusable experience into Skills. Compared with endlessly expanding the System Prompt, a Skill is a better carrier for a task class's steps, checklists, tool-usage advice, typical examples and result-interpretation conventions.

But a Skill should not decide:

- whether the user has permission to execute a control action;
- whether SQL is read-only;
- whether files may be written outside the Workspace;
- how much a temperature or frequency may be adjusted;
- whether an action requires human approval.

These constraints must be enforced by Tools, Policy and the Runtime. A Skill can tell the Agent "how to do it professionally," but it cannot decide "whether it is allowed."

### Three-level loading

```text
Level 1: name, description, when_to_use
Level 2: the full SKILL.md
Level 3: references, examples, scripts and templates
```

Level 1 can enter the Context as a lightweight catalog; the Agent judges relevance and calls `load_skill` to fetch the full instructions; only when execution reaches a concrete step does it read the reference files. This preserves discoverability without stuffing all professional knowledge into the context at once.

### User-triggered vs. Agent-discovered

I now divide Skill triggering into four kinds:

1. **Explicit user trigger**: the user picks a Skill or uses a slash command, and it loads directly this turn;
2. **Proactive Agent trigger**: the user's goal clearly matches `when_to_use`, and the Agent loads it on its own;
3. **Post-exploration trigger**: the Agent first inspects files, schemas or anomaly records, identifies the task type, then loads;
4. **Runtime trigger**: scheduled inspections, alerts or event tasks have the scheduler assign the Skill and inputs directly.

Proactive discovery shouldn't rely only on guessing from a single user sentence. If the user just says "help me analyze this file," the Agent can first inspect the file type and fields; once it discovers a multi-station time series, it loads the time-series-analysis Skill. This process is more stable than handing every Skill to the model up front.

## 5. Context Engineering is not about filling the window

After building several Agents, I increasingly agree with one judgment: the context window is not memory — it is the model's attention budget for this turn.

A 128K window does not mean you should put 128K of content in it. Tool descriptions, message history, Skills, file contents, database results and execution logs may all be useful, but when they coexist, the model finds it harder and harder to tell what matters most.

I now tend to split Context into six layers:

```text
1. Stable system rules: identity, boundaries, truthfulness and safety principles
2. Capability catalog: this turn's tool schemas, Skill metadata, permissions
3. Current task: user goal, attachments, todos, open questions
4. Project assets: relevant datasets, models, Artifacts and schema summaries
5. Working memory: recent messages, key tool results, current execution step
6. Compressed memory: past decisions, artifact references, failed attempts, unfinished items
```

Large files, full query results and long logs should live in the Workspace or object storage; the model receives only high-signal summaries and references it can read further. A tool returning "which file the result was saved to, what the key fields are, and how to filter next" is usually more useful than returning tens of thousands of lines of raw JSON.

### State cannot live only in Context

Context is what the model currently sees, not the system's reliable state. In-flight todos, approvals, generated assets, task progress, user questions and checkpoints should all be persisted to a database or event log.

This distinction is sharpest in human approval. If an approval only waits on an in-memory future while the original SSE connection has already dropped, the Agent cannot continue even if the user later clicks approve. The more reliable approach is to persist the approval record, the current message, the pending tools, the completed tool results and the current position, then resume from the checkpoint on a new request.

### Compaction has protocol boundaries too

You must not compact history after the model has emitted a tool call but before the tool result has been fully written back. Compaction should happen only after a complete tool-call protocol round, and it should preserve the user goal, key facts, artifacts, failure reasons, pending approvals and open questions — rather than producing a vague "conversation summary."

## 6. Rule-driven System Reminders are a valuable middle layer

Beyond Context Engineering, I've been paying more and more attention to a lighter mechanism: **the Runtime dynamically injects a short-lived reminder based on current events and rules, steering the Agent toward whatever is most error-prone right now.** I call it a System Reminder.

It is neither a System Prompt written in stone from the start nor a full domain Skill. The System Prompt describes long-term stable working principles; a Skill describes how to complete a class of tasks; a System Reminder answers exactly one question: **at this moment, what information must be pushed back to the front of attention?**

Claude Code gave me a lot of inspiration. Among its public mechanisms, [`CLAUDE.md` and Auto Memory](https://code.claude.com/docs/en/memory) provide persistent project context, while [Hooks](https://code.claude.com/docs/en/hooks) can return `additionalContext` or `systemMessage` when specific events occur. The exact implementation doesn't have to be copied, but this "event occurs — rule matches — context injected" pattern fits very well inside a general Agent Runtime.

I now distinguish these capabilities like this:

| Mechanism | Problem it solves | Typical lifetime |
| --- | --- | --- |
| System Prompt | Who the Agent is, what principles it follows long-term | Every model call |
| Skill | How a class of tasks should be completed | Loaded on demand during a task |
| System Reminder | What needs the most attention in the current state | Next call or current Run |
| Tool Result | What fact the outside world just returned | Current tool turn |
| Policy / Guardrail | Which behaviors must never be crossed no matter what the model decides | Enforced by the Runtime |

### Reminders should be event-triggered, not random interruptions

System Reminders are best for prompts that are **rule-clear, timing-critical and short-lived**. For example:

```text
skill_selected           user explicitly chose a Skill — remind to load its instructions before acting
capability_changed       tools or permissions changed — remind to re-confirm available capabilities
context_compacted        context was just compacted — restate the current goal, constraints and open items
tool_result_truncated    tool result was truncated — remind to continue paging/retrieving via the reference
tool_failed_repeatedly   same tool failed repeatedly — remind to stop mechanical retries and re-diagnose
approval_resumed         resumed after approval — remind not to re-run already-completed tools
workspace_file_uploaded  new asset entered the Workspace — remind to identify its type and trustworthiness first
context_budget_warning   attention budget running low — remind to save state and compact non-critical content
```

Take approval resumption: the Reminder can tell the model plainly, "of the three earlier tools the first two have completed; continue only with the third, still-pending action, and do not repeat the previous writes." Buried in a long history, this kind of information is easy to miss; written permanently into the System Prompt, it would be meaningless. It naturally belongs to event-triggered, temporary context.

### Generated by rules, not by piling up more prompt

These Reminders are best as declarative rules, generated by the Runtime from trusted state — not improvised by the model itself:

```yaml
id: resume-after-approval
trigger:
  event: approval_resumed
condition:
  pending_tool_count: "> 0"
scope: next_model_call
priority: high
dedupe_key: approval_id
content_template: |
  This Run resumed from an approval point. Completed tool results
  remain valid; continue only with the action pointed to by
  pending_tool_index, and never repeat earlier side-effecting operations.
```

A manageable Reminder should at least have a source, a trigger event, a condition, a priority, a scope, a dedup key and an expiry. When the Runtime builds the next round of Context, it injects the few matched high-priority reminders as a separate block; once consumed or once their condition disappears, they are removed immediately rather than sedimenting into the long-running conversation.

This is far more maintainable than "every time a mistake is found, append one more sentence to the System Prompt." Every rule is traceable: which event triggered it, in which Run it was injected, whether the model followed it, whether it actually reduced the error rate. That also makes it a natural fit for tracing and evaluation.

### System Reminders cannot replace hard constraints

A Reminder still only influences the model's attention; it provides no deterministic guarantee. A database tool that only allows `SELECT`, safety thresholds in industrial control, approval for dangerous actions, file-access scope and tenant isolation must all be enforced by the Policy Engine, state machines and tool code — never by a "please do not violate this" sentence.

At the same time, Reminder content must come only from trusted Runtime rules and verified state. Even if a web page, a user-uploaded document, database text or an ordinary Tool Result contains "ignore previous instructions," it must never be promoted into a system-level reminder — otherwise you have simply added a high-privilege channel for prompt injection.

So my ideal Context is not an ever-growing super-prompt, but:

```text
a stable System Prompt
  + on-demand Skills
  + ground truth from tools
  + resumable Run State / Memory
  + rule-driven, short-lived System Reminders
```

The value of System Reminders is not in adding more rules, but in letting the Runtime, at the right moment, lift the rules and state that already exist — and are most critical right now — back in front of the model's eyes.

## 7. What a standard Agent Runtime should own

As the projects gradually became products, I came to think the Runtime should take on at least these responsibilities:

```text
Model Gateway       models, providers, base URLs, context windows, streaming protocol
Agent Loop          LLM calls, tool loop, stop conditions, cancellation, max iterations
Tool Registry       schemas, discovery, execution, error normalization, result truncation
Toolset Resolver    selects this turn's tools by permissions, project and Skill
Context Builder     assembles rules, history, state, assets and user input
Reminder Engine     listens for events, matches rules, dedupes and injects short-lived reminders
Compactor           protocol-safe context compaction
Session / Run Store sessions, runs, events, concurrency locks, idempotency
Checkpoint          recovery from approvals, questions, disconnects and interruptions
Policy Engine       permissions, side effects, risk, approvals, rate limits
Workspace / Sandbox file isolation, code execution, resource limits
Skill Registry      discovery, search, on-demand loading, versions, user scope
Asset Registry      datasets, models, reports, charts and Artifacts
Event System        SSE / WebSocket progress and events
Observability       traces, tokens, latency, tool errors, task outcomes
Evaluation          tool selection, parameter accuracy, task success rate, safety evaluation
```

Multi-Agent is not a required feature of a Runtime. Many tasks are well served by one main Agent plus a set of well-designed tools. Constrained Sub-Agents are only worth adding when subtasks can genuinely run in parallel, need to isolate large amounts of context, or need independent permission boundaries.

## 8. The System Prompt is the constitution

I've lived through the System Prompt growing longer and longer in projects: the model repeats queries, so add a paragraph banning retries; the wrong tool gets picked, so add a tool-selection table; report format is unstable, so add a report template; plots look bad, so add plotting code.

In the short term, each addition fixes one problem; in the long term, it becomes a mixture of tool documentation, business rules, failure branches, output templates and visual guidelines. Rules start to conflict, and changing one place makes it hard to tell which tasks will be affected.

Now I prefer to keep only stable principles in the System Prompt:

```text
<identity>role and target users</identity>
<scope>what it may do and may not do</scope>
<truth_policy>must rely on real tool results</truth_policy>
<execution_policy>how to explore, act, check and stop</execution_policy>
<tool_policy>general selection principles across tools</tool_policy>
<skill_policy>when to load Skills, and instruction precedence</skill_policy>
<workspace_policy>rules for input, intermediate and output directories</workspace_policy>
<safety_policy>permissions, sensitive data and dangerous-operation principles</safety_policy>
<completion_contract>how completion, failure and artifacts are reported to the user</completion_contract>
```

Each tool's parameters live in the Tool Schema, domain procedures live in Skills, strict constraints live in code, project assets are injected through dynamic Context, and long-form reference knowledge is fetched through retrieval. The System Prompt is only responsible for telling the Agent who it is, how it works, and which boundaries cannot be crossed.

Reminders that only matter in specific states are left to the Reminder Engine to inject dynamically, so the global Prompt no longer keeps swelling for the sake of one local failure scenario.

## 9. A few anti-patterns I kept stepping on

### Fixing every problem with prompts

If the problem is fundamentally parameter validation, permissions, timeouts, retries or state recovery, adding more prompt usually just postpones the fix.

### Turning low-level APIs into tools as-is

Human developers easily understand the relationships among dozens of resource APIs; the model may not. Agent Tools should be task-oriented, not merely API-oriented.

### Preloading the full text of every Skill

This wastes Context and lets multiple domain procedures interfere with each other. Preloading metadata and reading the body on demand is more sensible.

### Letting tools return too much raw data

Large results should be saved and returned by reference; the model's context keeps only what this turn's decision needs.

### Treating "code exists" as "capability online"

A Tool class, approval module or CLI being written doesn't mean the current entry point has registered it, nor that the user has permission to use it. A Capability Registry should explicitly distinguish: implemented, configured, currently enabled, available to the current user.

### Using Multi-Agent to paper over tool and context problems

If a single Agent hasn't even nailed down tool boundaries, state and completion conditions, splitting into multiple Agents usually just adds communication and debugging cost.

## 10. The Agent shape I currently endorse

If I redesigned a general-purpose Agent from scratch today, I would default to:

```text
one main Agent
  + a small, clear set of base tools
  + domain tools loaded dynamically per project
  + Level 1/2/3 progressive Skills
  + project-level Workspace and asset management
  + event-sourced Run State and Checkpoints
  + rule-driven short-lived System Reminders
  + Tool/Action-level permissions and approvals
  + a controlled sandbox and real-result verification
```

High-risk, highly deterministic processes — industrial control, for example — are not handed to the model to orchestrate freely. The model is responsible for understanding intent, gathering evidence and generating recommendations; actual execution still goes through deterministic state machines, safety checks and human approval. Open-ended data analysis, code execution and research tasks, on the other hand, can give the Agent more autonomy inside the sandbox.

After finishing these projects, my understanding of Agents has shifted from "can we make the model call tools" to a different question: **how to let the model, within a limited context and clear boundaries, continuously receive real feedback — and turn an uncertain generation process into an observable, resumable, verifiable system execution.**

That may also be the part of Agent engineering I currently consider most worth accumulating over the long term.
