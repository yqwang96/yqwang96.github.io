Many "let an LLM do operations research" setups eventually collapse into a deceptively simple interface: feed in a problem, get back a number. But the hard part is rarely that final line of output — it is everything in the middle: are the variables defined correctly, are any constraints missing, does the code even run, does the solver actually find the optimum, and is the final answer faithful to the solved result.

OR-Agent-R1 is my end-to-end take on that pipeline: the model turns a natural-language OR problem into structured actions, then emits executable PuLP code that is run by the CBC solver, and the environment returns a reward based on the execution outcome and whether the objective matches.

## 1. What problem is this project trying to solve?

Traditional QA-style evaluation only checks whether the final answer is correct. OR-Agent-R1 reframes the task as an interactive agent episode:

![The verifiable agent loop of OR-Agent-R1](/images/Blog24Fig1.svg)

The model has to carry out extraction, mathematical formulation, code generation, solving, and revision when needed. The action protocol uses XML tags, with core actions `<extract>`, `<formulate>`, `<solve>`, `<revise>`, and `<answer>`. Among these, `<solve>` and `<revise>` trigger the executor, so the environment can observe whether the code runs, whether the solver status is `optimal`, and whether the objective matches the reference answer.

This shifts the training objective from "produce text that looks plausible" to "complete a decision process that a tool can verify."

## 2. Why isn't a final-answer reward enough?

With only a final 0/1 answer reward, a single episode can fail in many different ways: malformed XML, wrong action order, a generated placeholder, code that won't execute, wrong variable types, mis-built constraints, or a solver that succeeds but on the wrong objective. Every failure collapses into the same 0, so the model can hardly tell what to fix next time.

That is why the project splits the reward into several observable layers:

![From a sparse outcome to dense, learnable signals](/images/Blog24Fig2.svg)

```text
format correct
  -> state transition correct
  -> produces non-empty code
  -> code is executable
  -> solver returns optimal
  -> objective matches the reference answer
```

Training follows a `solve_format -> solver -> objective` curriculum. The model first learns to reliably emit legal tool actions, then the reward is gradually wired to the real solver, and finally the correctness of the math result is reinforced. The point of this ordering is not to make the reward more complicated, but to let each stage answer one concrete question: did the agent reach the right tool entry point, did the tool actually run, and did the run actually complete the task?

## 3. Where engineering is easiest to misjudge

Early experiments once produced almost all-zero results. Only after checking item by item did it become clear that the problem was not entirely model capability:

- the executor accepted only one PuLP import style and failed to recognize the model's common `from pulp import *`;
- when the code used `prob` or `problem`, the executor only looked for `model`;
- the runtime was missing PuLP, so every program hit a runtime error before any real solving;
- placeholders like `<solve>...</solve>` were not penalized separately;
- numeric answers had rounding and truncation, and a verifier tolerance that was too strict would mark reasonable solutions as wrong.

The lesson is that in agent RL, the environment, executor, and verifier are themselves part of the training system. Before training, you have to run canary tests on the parser, state machine, code extraction, solver, reward, and trace logging — otherwise "the model can't learn" and "the environment never let it learn" get easily confounded.

## 4. Results: where does the gain come from?

The strict main experiment uses a fixed train/validation/held-out split: 1227 training samples, 153 validation samples, and 154 final test samples. The checkpoint is selected using validation only, and the held-out set is evaluated exactly once at the end.

At a tolerance of `5e-3`, the Qwen2.5-7B-Instruct base model reaches a pass@1 of **64.29%**, and the LoRA + GRPO RLVR adapter lifts it to **74.03%**, an absolute gain of **9.74 points**.

![Qwen2.5-7B: pass@1 on the strictly held-out set](/images/Blog24Fig3.svg)

This gain is not simply "training reward went up." A paired, per-sample analysis shows that LoRA fixed 34 samples the base model failed on, while regressing 19 that used to succeed — a net gain of 15/154 — and a McNemar exact test gives `p=0.0275`. That is why I prefer reporting the paired result rather than just comparing two independent averages.

## 5. What actually changed behind the numbers

In a separate development-stage comparison, the RL LoRA's invalid action rate dropped from 20.4% to 4.7%, the average number of agent actions fell from 5.49 to 2.805, and the solver success rate rose from 89.0% to 92.5%. This suggests the gains come mainly from three kinds of change:

1. **More stable actions**: the model emits far fewer actions the environment cannot accept.
2. **A shorter tool chain**: successful trajectories usually do not need repeated retries and reach the solver faster.
3. **Some modeling errors get fixed**: a solver returning `optimal` still does not mean the formulation is correct; the objective match is the final check.

One caveat: the development-stage 200-sample comparison was drawn from the training pool at the time and should not be treated as strict generalization evidence; the final conclusions rest on the isolated held-out experiment.

## 6. A few takeaways from this project

### The verifier sets the ceiling of RLVR

If the reference answer's precision, variable types, solver status, or final-answer extraction has a systematic bias, the model will learn that bias very reliably. RLVR does not automatically repair a broken verifier.

### Correct tool use is not the same as a correct task

Evaluation should at least be split into protocol correctness, code executability, solver success rate, modeling correctness, and final-answer consistency. Reporting only one of these hides the real bottleneck.

### Small models can also benefit from a high-quality environment

This experiment uses Qwen2.5-7B, not a very large model. The results suggest that on verifiable tasks, getting the execution environment, reward chain, and evaluation methodology solid first often matters more than simply adding training steps.

## 7. What's next

Three directions are still worth pursuing: a systematic error analysis of the `base_only` and `both_fail` samples; reproducing the training gains across multiple random seeds; and a fuller comparison of the full curriculum against an answer-only reward, to confirm the stability of a sparse final reward over longer training.

The code and experiment scripts are organized in [OR-Agent-R1](https://github.com/yqwang96/OR-Agent-R1). For me, the most important output of this project is not any single pass@1 number, but a way to connect "language model generation" to "executable optimization decisions."
